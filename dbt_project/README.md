# Модуль трансформации данных dbt (Data Build Tool)

Этот подпроект отвечает за слой трансформации, стандартизации и проверки качества данных маркетплейса Wildberries перед их отправкой в BI-слой.

## Назначение слоя
Использование dbt позволило инкапсулировать правила обработки данных маркетплейса внутри SQL-кода, изолировать сырые данные от аналитических моделей и внедрить автоматическое тестирование ключевых бизнес-полей.

---

## Структура слоев данных

### 1. Слой `staging`
Первичная обработка сырых сущностей Wildberries. Здесь происходит переименование полей в читаемый технический вид, явное приведение типов (даты, таймстампы, денежные типы) и первичная фильтрация некорректных строк.

* `stg_orders` — сырые данные по созданным заказам.
* `stg_sales` — первичные факты выкупов.
* `stg_sales_report` — сырые строки бухгалтерских отчетов о реализации маркетплейса.
* `stg_stock` — ежедневные логи остатков товаров.
* `stg_supplies` — данные о планируемых и принятых поставках.

### 2. Слой `marts`
Финальные денормализованные витрины данных, спроектированные для прямого подключения к схеме «Звезда» в BI-системах.

* `v_orders` — история заказов со статусами отмен для построения воронки.
* `v_sales` — чистые факты продаж для оперативного отслеживания выручки и географии.
* `v_report_realization` — финансовая витрина на основе отчетов реализации (расчет удержания комиссий WB, стоимости логистики и чистой выручки к перечислению).
* `v_stock` — витрина остатков и оборачиваемости товаров, включая позиции в пути до и от клиента.
* `v_supplies` — аналитика принятых объемов поставок на целевые склады.

---

## Тестирование и качество данных
В проекте настроено декларативное тестирование моделей через `schema.yml`. 
* На уровне `staging` и `marts` проверяется обязательное отсутствие `NULL`-значений (`not_null`) в критических бизнес-ключах (`odid`, `rrd_id`).
* Конфигурация dbt готова к масштабированию тестов уникальности комбинаций полей для исключения дублирования строк при повторных выгрузках данных маркетплейса.
